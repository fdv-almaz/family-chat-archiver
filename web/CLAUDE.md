# CLAUDE.md — Web интерфейс

Веб-приложение для просмотра, поиска и управления данными, накопленными ботом.

## Структура

```
web/
├── app.py            # FastAPI routes
├── config.py         # env переменные
├── db.py             # SQL-запросы к MySQL (read-only + delete)
├── telegram.py       # скачивание медиа из Telegram с кешированием
├── templates/        # Jinja2: index, message, users, stats, corrections, tips
├── static/           # style.css, app.js (бар-чарт без зависимостей)
│                      # favicon — inline SVG-эмодзи в base.html, файла нет
├── media_cache/      # локальный кеш скачанных файлов
├── requirements.txt
├── .env.example
└── CLAUDE.md
```

## Запуск

```bash
cd web
pip install -r requirements.txt
cp .env.example .env
# Отредактировать MYSQL_*, TELEGRAM_BOT_TOKEN (для медиа)
python app.py
# Открыть http://127.0.0.1:8000
```

## Конфигурация (.env)

| Переменная | По умолчанию | Назначение |
|-----------|--------------|------------|
| `WEB_HOST` | `127.0.0.1` | Интерфейс биндинга. `0.0.0.0` — все интерфейсы |
| `WEB_PORT` | `8000` | Порт биндинга |
| `MYSQL_*` | — | Доступ к общей с ботом БД |
| `TELEGRAM_BOT_TOKEN` | — | Только для fallback-скачивания медиа |
| `MEDIA_CACHE_DIR` | `media_cache` | Кеш fallback-файлов |
| `ALLOWED_MEDIA_DIRS` | storage-папки + cache | Whitelist путей для `/media/{id}` |
| `ALLOWED_ORIGINS` | — | Доп. Origin/Referer для POST/DELETE (CSRF) |

`WEB_HOST`/`WEB_PORT` читаются в `config.py` и применяются в
`app.py::uvicorn.run()`. При старте адрес биндинга логируется; если
`WEB_HOST` не loopback (`127.0.0.1`/`localhost`/`::1`) — выводится
предупреждение об отсутствии авторизации.

## Возможности

- **`/`** — список сообщений
  - Фильтры: чат (имя, не ID), пользователь, тип, даты, полнотекстовый поиск
  - Пагинация: размер страницы 25–1000 с шагом 25, по умолчанию 100
  - Глобальная нумерация строк (продолжается между страницами)
  - **Инлайн-плеер** для audio/voice, миниатюры для photo/video, ссылка для documents — слушать/смотреть можно сразу из списка
  - Маркер удалённых сообщений (красноватый фон, зачёркнутый текст, иконка)
  - Маркер орфо-исправления: ✏️ перед текстом, если сообщение правил спеллчекер; в подсказке при наведении (`title`) — пары «было → стало». Источник — `spelling_corrections` (см. `db.list_messages` → `spelling_hint`, хелпер `_correction_pairs`)
- **`/message/{id}`** — карточка с воспроизведением медиа, ссылками, орфо-подсказками
  - Кнопки: soft-delete / restore / hard-delete
- **`/users`** — пользователи + количество сообщений
- **`/stats`** — карточки счётчиков, бар-чарт за 30 дней, топ-10 авторов, типы сообщений
- **`/corrections`** — лента орфографических исправлений
- **`/tips`** — лента «советов дня» из таблицы `daily_tips` (когда, чат, модель, текст совета, статус отправки). В общую ленту сообщений советы не попадают — бот игнорирует собственные сообщения, — поэтому для них отдельная страница
- **`/media/{media_id}`** — стриминг файла из Telegram через локальный кеш

## Удаление сообщений (soft-delete)

- POST `/message/{id}/delete` — выставляет `messages.deleted_at = NOW()`
- POST `/message/{id}/restore` — снимает пометку (NULL)
- POST `/message/{id}/hard-delete` — физическое удаление (с каскадом по media/links/corrections)

Колонка `deleted_at` добавляется автоматически при старте веб-приложения (см. `_ensure_deleted_at_column` в `db.py`).

⚠️ Telegram Bot API не уведомляет ботов об удалении сообщений в чате. Если хотите автодетект — нужен отдельный MTProto-companion на Telethon с user-сессией.

## Как работают медиа

Приоритет источников в `/media/{id}`:

1. **`media.local_path`** — файл, скачанный ботом в `python/storage/` (или `rust/storage/`) при получении сообщения. Это основной путь — Telegram не дёргается вообще.
2. **Fallback: скачивание из Telegram** — если `local_path` пуст (legacy данные) или файл с диска пропал:
   - `getFile` → file_path
   - Скачивание из `https://api.telegram.org/file/...`
   - Сохранение в `media_cache/{file_unique_id}{ext}`

⚠️ `file_id` живёт ~24 часа. Если ботов давно не было, fallback может вернуть 502. Поэтому `local_path` от бота — надёжнее.

> Если хотите, чтобы веб читал прямо из папки бота, можно подмонтировать `python/storage/` симлинком в нужное место или указать абсолютный путь в `MEDIA_STORAGE_DIR` бота, который будет виден веб-процессу. Сейчас `local_path` сохраняется как абсолютный путь, рассчитанный ботом — веб открывает его напрямую.

## Безопасность

> Раздел поддерживается актуальным по итогам security-аудита 0.9.1.

**Что уже сделано в коде:**
- ✅ **SQL injection** — все пользовательские запросы параметризованы (`%s`). DDL (`ALTER TABLE`) — только с хардкод-именами колонок.
- ✅ **Path traversal в `/media/{id}`** — `local_path` из БД проверяется на принадлежность `ALLOWED_MEDIA_DIRS` (по умолчанию `web/media_cache`, `../python/storage`, `../rust/storage`). Любой путь вне whitelist → 403.
- ✅ **XSS** — Jinja2 `autoescape = True` форсированно включён в `app.py`, любой текст из БД экранируется.
- ✅ **CSRF** — middleware `same_origin_for_unsafe_methods` блокирует POST/DELETE с чужим `Origin`/`Referer`. Дополнительные источники — через `ALLOWED_ORIGINS` в `.env`.
- ✅ **DoS через `q`** — `Query(None, max_length=200)`.
- ✅ **Лимит размера медиа** — бот пропускает > 20 МБ (Bot API hard limit).

**Что должен сделать оператор:**
- 🔒 **Авторизация отсутствует** — слушать только `127.0.0.1` (`WEB_HOST=127.0.0.1` по умолчанию). Для удалённого доступа поставить **nginx + Basic Auth / OAuth** и HTTPS.
- 🔒 **MySQL пользователь** — отдельная учётка с правами только на конкретную БД, не root.
- 🔒 **`.env` не пушить** — он в `.gitignore`; права `chmod 600`.
- 🔒 **Логи** — могут содержать тексты сообщений; ограничьте доступ к `logs/`.

**Что не закрыто (известные ограничения):**
- Rate-limiting отсутствует. При публичном размещении ставьте за reverse-proxy с лимитом запросов.
- `page_size` до 1000 — может дать нагрузку на БД при злоупотреблении.

## Связь с ботом

- Использует **ту же MySQL базу** что и Python/Rust бот
- `TELEGRAM_BOT_TOKEN` нужен только для скачивания медиа; updates веб НЕ потребляет — конфликта 409 с ботом не будет
- Бот может работать одновременно

## Расширение

Что можно добавить дальше:
- Авторизация (Basic Auth, Telegram OAuth)
- Экспорт сообщений (JSON, CSV)
- Удаление по фильтру (батч-операции)
- Просмотр служебных событий
- Полнотекстовый индекс (MySQL FULLTEXT или Meilisearch)
- Real-time через WebSocket: новые сообщения сразу появляются в UI
