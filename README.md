# Family Chat Archiver

**Версия 0.10.1** — см. [CHANGELOG.md](CHANGELOG.md) · [Авторы](CONTRIBUTORS.md)

Телеграм-бот для архивирования всех сообщений семейной группы с автоматической проверкой правописания русскоязычных текстов, плюс веб-интерфейс для просмотра и управления архивом.

## Основные возможности

✅ Сохранение всех сообщений (текст, фото, видео, аудио, голос, стикеры, документы, и др.)
✅ Денормализованные имена авторов и названия чатов прямо в сообщениях
✅ Сохранение ссылок и служебных событий (user_joined, title_changed и т.д.)
✅ Проверка орфографии через YandexSpeller API (настраиваемая видимость: public/private/off)
✅ «Совет дня» от модели Claude — ежедневная рассылка в чат по расписанию (учитывает, что в чате есть и дети, и взрослые) + команда `/check_tip`
✅ Веб-интерфейс: просмотр, фильтры, полнотекстовый поиск, воспроизведение медиа, статистика, лента советов дня, soft-delete
✅ Файловые логи с ежедневной ротацией
✅ Две реализации бота: Python (быстрая разработка) и Rust (production)

## Быстрый старт

### Предусловия

- MySQL 8.0+
- Python 3.8+ (для Python версии)
- Rust 1.70+ (для Rust версии)
- Telegram Bot Token (получить от @BotFather)

### Создание БД (один раз, общее для обеих реализаций)

```bash
# Схема создаётся автоматически при первом запуске бота, но можно явно:
mysql -u root -p < schema.sql
```

### Python версия

```bash
cd python

# Установка зависимостей
pip install -r requirements.txt

# Создание конфига
cp .env.example .env
# Отредактировать .env с вашими данными (TELEGRAM_BOT_TOKEN, MYSQL_*, SPELLING_VISIBILITY)

# Запуск бота (таблицы создаются автоматически)
python bot.py
```

### Веб-интерфейс (просмотр, поиск, медиа, статистика)

```bash
cd web

pip install -r requirements.txt
cp .env.example .env
# Отредактировать MYSQL_*, TELEGRAM_BOT_TOKEN (нужен для скачивания медиа)
# При необходимости задать WEB_HOST/WEB_PORT (по умолчанию 127.0.0.1:8000)

python app.py
# Откроется на http://127.0.0.1:8000
```

**Адрес и порт биндинга** настраиваются в `.env`:

```env
WEB_HOST=127.0.0.1   # 0.0.0.0 — слушать на всех интерфейсах
WEB_PORT=8000
```

⚠️ У веб-интерфейса нет встроенной авторизации. `WEB_HOST=0.0.0.0` (или
любой не-loopback адрес) выставляет архив наружу — допустимо только за
reverse-proxy (nginx) с auth и HTTPS. При старте на не-loopback адрес
бот пишет предупреждение в лог.

Веб НЕ конфликтует с ботом (не потребляет updates), может работать одновременно с Python/Rust версиями. Использует ту же MySQL.

## Конфигурация

Создайте `.env` файл в корне выбранного проекта:

```env
TELEGRAM_BOT_TOKEN=your_token_here
MYSQL_HOST=localhost
MYSQL_PORT=3306
MYSQL_USER=root
MYSQL_PASSWORD=your_password
MYSQL_DATABASE=family_chat

# Логи
LOG_LEVEL=info
LOG_FILE=logs/bot.log           # путь файла логов (Python)
LOG_RETENTION_DAYS=7            # дней хранить ротированные логи (Python)
LOG_TO_CONSOLE=true             # дублировать в stdout
# В Rust версии: LOG_DIR=logs, LOG_FILE_PREFIX=bot.log

# Видимость орфографических подсказок:
#   public  - reply в чате, видят все
#   private - личное сообщение автору (он должен сначала /start бота в DM)
#   off     - в чат не отправлять, только сохранять в БД
SPELLING_VISIBILITY=public

# Совет дня (включается заданием ANTHROPIC_API_KEY; без него фича выключена)
ANTHROPIC_API_KEY=your_anthropic_key   # ключ Anthropic API
ANTHROPIC_MODEL=claude-opus-4-8        # модель Claude для совета
TIP_CHAT_ID=                           # чат рассылки; пусто → самый активный групповой чат из БД
TIP_HOUR=6                             # время рассылки (локальное)
TIP_MINUTE=0
```

«Совет дня» работает как фоновый планировщик внутри самого бота (отдельные программы/cron не нужны): ежедневно в `TIP_HOUR:TIP_MINUTE` бот запрашивает у модели Claude короткий совет и шлёт его в чат, сохраняя запрос и ответ в таблицу `daily_tips`. Команда `/check_tip` (или `/tip`) генерирует совет немедленно и отправляет в текущий чат — удобно для проверки.

## Схема базы данных

Все таблицы автоматически создаются при первом запуске (см. `schema.sql`):

- **users** — информация о пользователях
- **messages** — архив сообщений (с денормализованными именами автора/чата, столбец `deleted_at` для soft-delete из веб-интерфейса)
- **media** — информация о медиа-файлах (photo, video, audio, voice, video_note, animation, document, sticker)
- **links** — ссылки из сообщений
- **spelling_corrections** — история орфографических ошибок и исправлений
- **service_events** — служебные события группы (вход/выход, изменение названия и т.д.)
- **daily_tips** — история «совета дня» (запрос, ответ, модель, статус отправки)

> **Примечание о soft-delete:** колонка `deleted_at` добавляется автоматически при запуске веб-интерфейса. Telegram Bot API **не уведомляет** ботов об удалении сообщений в чате, поэтому пометка `deleted_at` сейчас ставится только вручную через веб-UI. Для автоматического отслеживания удалений в чате потребовался бы отдельный сервис на базе MTProto (Telethon) с пользовательской сессией.

## Различия между реализациями

Обе версии поддерживают тот же набор фич и пишут в одну MySQL. Запускать одновременно с одним токеном **нельзя** (Telegram отдаёт updates только одному getUpdates).

### Python
- **Плюсы:** быстрая итерация, простой запуск
- **Использование:** основная рабочая версия

### Rust
- **Плюсы:** минимальное потребление памяти, типобезопасность, async/tokio
- **Использование:** альтернатива для долгоживущих процессов / слабых машин

### Web
- Не требует выбора между Python/Rust — работает с базой обоих
- Не потребляет Telegram updates (нет конфликта 409)

## Решение проблем

### Бот не получает обновления

1. Проверьте TELEGRAM_BOT_TOKEN
2. Убедитесь, что нет других ботов с тем же токеном
3. Проверьте сетевое соединение
4. Изучите логи

### Ошибки БД

1. Убедитесь, что MySQL запущена
2. Проверьте credentials в .env
3. Убедитесь, что у пользователя есть права на создание таблиц

### YandexSpeller не работает

1. Проверьте подключение к интернету
2. YandexSpeller API может быть перегружена, есть retry logic
3. Проверьте логи для деталей ошибок

## Безопасность

Проект прошёл внутренний security-аудит (0.9.1). Внутри кода закрыто:

- **SQL injection** — все запросы параметризованы.
- **Path traversal** — `web/media/{id}` отдаёт файлы только из whitelist
  (`ALLOWED_MEDIA_DIRS`).
- **XSS** — Jinja2 autoescape принудительно включён.
- **CSRF** — same-origin guard для POST/DELETE (см. `ALLOWED_ORIGINS`).
- **DoS через поиск** — `q` ограничен 200 символами.
- **Лимит скачивания медиа** — 20 МБ (лимит Bot API).
- **Секреты** — `.env` в `.gitignore`, токены не пишутся в логи.

Что обязан обеспечить оператор:

- 🔒 **Веб-интерфейс не имеет встроенной авторизации.** Слушает
  `127.0.0.1` по умолчанию. Для удалённого доступа — только за
  reverse-proxy (nginx) с Basic Auth/OAuth и HTTPS.
- 🔒 Отдельный MySQL-пользователь с правами только на БД проекта.
- 🔒 Доступ к `logs/` и `storage/` ограничить — там тексты сообщений
  и медиа-файлы.
- 🔒 Запускать Python и Rust бота с **разными** токенами @BotFather,
  если нужны параллельно.

Подробности — в `web/CLAUDE.md` раздел «Безопасность».
